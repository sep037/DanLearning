## TCP와 UDP
### 1. TCP의 특징을 설명해 보세요.

- 송신부와 수신부의 연결을 확인하는 연결형 서비스

- 가상회선방식(패켓이 전달되는 회선이 정해져있음)

- 패킷 전송 순서 보장

- 송신부, 수신부 1:1 통신


- 높은 신뢰성(데이터손실없음 보장)


- 데이터 송수신 속도 느림

------

### 2. TCP의 연결 및 해제 과정을 설명해 보세요.
- 연결 : 3-way handshaking
  
  1. 송신부가 수신부와 연결하기 위해 임의의 숫자 N과 SYN 메세지 보냄. 송신부는 응답이 오기전까지 SYN_SENT상태
  2. 수신부는 SYN메세지를 받고 N+1값과 연결요청을 수락하는 ACK 메세지를 전송. SYN메세지에 임의의 숫자 M을 보내고 응답을 기다림. 수신부는 SYN_RECEIVED 상태.
  3. 송신부가 수신부로부터 ACK+SYN 메세지를 받으면 연결성립을 의미하는 ESTABLISHED 상태가 됨. ACK 메세지와 M+1값을 보냄 (이떄 ACK 메세지에 송신부에서 전송데이터 포함될 수 있음). 송신부로부터 ACK 메세지를 받으면 수신부는ESTABLISHED 상태.
 
- 해제 : 4-way handshaking

  1. 송신부가 수신부와의 연결을 종료하려고 FIN 메세지를 보냄. 이떄 송신부는 FIN_WAIT1 상태.
  2. 수신부가 FIN메세지를 받고 응답으로 ACK 메세지 보냄.이떄 수신부는 CLOSE_WAIT 상태. 메세지를 보낸후 수시부는 연결 종료하기 위한 작업. 송신부는 수신부에서 보낸 ACK 메세지 받고 FIN_WAIT2 상태가 됨.
  3. 수신부에서 연결을 종료할 준비가 끝나면 송신부에게 FIN 메세지를 보내고 LAST_WAIT 상태가 됨.
  4. 송신부는 서버로부터 받은 FIN 메세지에 응답하기 위해 ACK 메세지를 보내고 TIME_WAIT 상태가 됨. 일정시간 후 CLOSED 상태.
       + 일정시간 동안 TIME_WAIT 상태 유지 이유:
          1. FIN메세지 전에 보낸 패킷이 FIN메세지 수신보다 지연되어 발생하는 패킷 유실 대비.
          2. 수신부에게 ACK 메세지가 제대로 전달되지 않아 연결 해제가 이뤄지지 않는 경우 대비

### 3. TCP에서 데이터의 신뢰성을 보장하기 위한 방법을 설명해 보세요.
- 흐름제어 : 데이터 송신부와 수신부에서 데이터 처리속도의 차이 때문에 생기는 데이터 손실 방지

  1. 정지대기 : 송신부에서 데이터를 보낸 후 수신부로부터 ACK 메세지를 받을때까지 다음 데이터를 보내지 않고 기다림
      - 일정시간 동안 수신부로부터 ACK 메세지를 받지 못하면 패킷 재전송
      - 시간면에서 비효율


  2. 슬라이딩 윈도우 : 송신부에서 ACK 확인하지않고 수신부에서 설정한 윈도우 크기만큼 데이터를 연속적으로 보낼 수 있게 해서 데이터 흐름을 동적으로 제어
     - 윈도우 크기 : 응답받지 않고 보낼 수 있는 데이터의 최대개수, 3-way handshakinh에서 정해짐
     - n~n+N-1까지 보냈을때 n번에 대한 ACK를 받으면 n+N번 데이터 전송 가능-> 윈도우가 한칸이동하는 것
     - 정지 대기방식의 단점 보완 (수신 여부와 상관없이 일정 크기의 데이터를 연속적으로 보냄)
 
 - 혼잡 제어 : 송신부의 데이터 전달 속도와 네트워크 속도 차이로 데이터 손실이 발생하는 것 방지

   
   1. AIMD (Additive Increase Multiplicative Decrease)
     - 데이터를 전달할때 합증가 방식으로 혼잡윈도우 크기를 더해가면서 키움
     - 데이터 손실이 발생하면 혼잡 윈도우 크기를 곱감소로 배수 단위로 줄임
     - 네트워크 대역폭을 공평하게 사용 가능
     - 데이터 유실 발생시 네트워크 대역폭을 넓게 사용하기까지 시간이 오래걸림
       
   2. 느린시작
    - 윈도우 크기가 1인 상태에서 시작, ACK 메세지를 수신할때마다 윈도우 크기를 +1
    - 혼잡 발생시 윈도우 크기를 1로 줄임
    - AIMD에서 초기에 전송 가능한 패킷 수가 적다는 단점 보완
  
      
   3. 혼잡 회피
    - 윈도우 크기가 지수함수 형태로 증가
    - 혼잡 발생 방지를 위해 윈도우 크기에 대한 임계점 정함
    - 임계점 도달시 선형적 증가
    - 타임아웃 발생 시점에서 윈도우 크기의 절반을 임계점으로 설정, 윈도우 크기를 초깃값으로 변경
  
      
   4. 빠른 회복 : 혼잡 발생시 혼잡 윈도우 크기를 절반으로 줄인후 선형적 증가
    - 혼잡 처음 발생시 AIMD 방식


   5. 빠른 재전송
    - Duplicate ACK 3번 발생시 해당 시점의 윈도우 크기를 1/2로 줄임
  
  - 오류 제어 : 데이터에 오류, 유실 발생할때 데이터의 신뢰성을 보장하기 위해 오류 제어하는 방식
      - 정지-대기 : 타임아웃 발생시 해당 패킷 재전송
      - Go-Back-N ARQ : 연속적 데이터 보냈을때 누락 데이터를 송신부에서 해당 데이터부터 재전송
      - Selective-Repeat ARQ : 연속적 데이터 보냈을때 누락 데이터 있을시 수신부에서 해당 데이터만 재전송 요청

### 4. UDP의 특징을 설명해 보세요.
- 송신부와 수신부 연결 보장x=비연결형 서비스
- 패킷이 서로 다른 회선으로 교환될 수 있는 데이터그램 패킷교환방식
- 송신부에서 전달한 패킷 순서와 수신부에서 받은 패킷 순서 다를 수 있음
- 패킷 수신여부 확인x
- 1:1, 1:N, N:N 통신 가능
- 데이터 신뢰성 낮음
- 데이터 전송속도 빠름

### 6. UDP에서 오류 검출 방법을 설명해 보세요.

체크섬 방식 사용
- 체크섬 : UDP 헤더, IP 헤더 일부와 데이터로 체크섬 값 생성
- 생성한 체크섬 값을 UDP 헤더의 체크섬 영역에 넣어서 수신부에 보냄
- 수신부는 체크섬을 포함한 모든값을 더해 비트가 1인지 확인
- 1=송신부와 동일한 체크섬임을 의미

  
- 단점
  - 데이터의 순서, 오류 발생시 검출 불가
  - 체크섬은 선택사항이므로 0으로 체크섬을 보내면 계산안함

## HTTP와 HTTPS
### 7. HTTP에 대해 설명해 보세요.
- 인터넷상에 데이터를 전송하기 위한 프로토콜, TCP/IP 4계층에 속함
- 특징
  - 비연결성 : 클라이언트에서 요청을 보낸후 서버로부터 응답을 받으면 연결끊음
  - 무상태 : 서버에서 클라이언트 상태 저장안함
    
----


### 8. HTTP Keep Alive와 TCP Keep Alive를 설명해 보세요.
- HTTP Keep Alive : HTTP 연결시 일정시간 동안 요청 유지할 수 있도록 사용하는 HTTP 헤더 일종
- 클라이언트에서 HTTP 요청 보낼때 연결 헤더애 Keep Alive 추가해서 보내면 서버에서 연결을 유지할 시간을 Keep Alive 헤더에 추가해 응답

---- 


### 9. 쿠키와 세션의 차이점을 설명해 보세요.
- 쿠키 : 클라이언트의 로컬 웹 브라우저에 저장하는 데이터 파일. 키와 값 저장
- 세션 : 서버에서 클라이언트와의 연결 정보를 저장 및 관리. 서버에 데이터가 저장된다는 점이 보안면에서 쿠키보다 좋음. 하지만 접속자가 많을 경우 과부화.


---- 

### 10. HTTP와 HTTPS의 차이점을 설명해 보세요.
- HTTPS : 보안 계층인 SSL/TLS를 이용해 HTTP의 보안을 강화한 웹 통신 프로토콜
- SSL : 넷스케이프에서 개발한 암호화 프로토콜
- TLS : SSL을 보완한 암호화 프로토콜
- HTTPS 동작방식 : 데이터를 송신할때 응용계층->보안계층(데이터암호화)->전송계층, 데이터를 수신할때 전송계층에서 보낸 데이터를 보안계층에서 복호화-> 응용계층

----


### 11. HTTPS에서 사용하는 암호화 방식을 설명해 보세요.
- 대칭 키 암호화 방식 : 데이터의 암호화, 복호화에 같은 키인 대칭키 이용
- 공개 키 암호화 방식 : 데이터의 암호화, 복호화에 다른 키 사용. 데이터암호화-공개키, 데이터복호화-비밀키

----


### 12. 사용자가 URL을 입력한 후 화면이 출력되기까지의 과정을 설명해 보세요.
1. 사용자가 URL을 웹브라우저에 입력
2. 웹브라우저는 URL을 바탕으로 DNS 서버에 연결할 IP 요청
3. DNS 서버는 IP 주소를 웹브라우저에게 응답으로 제공
4. 웹브라우저는 DNS 서버에ㅓ 받은 IP를 통해 웹서버와 TCP/IP 연결을 하고 HTTP 요청 보냄
5. 웹 서버는 받은 HTTP 요청에 응답( 웹페이지, 필요한 리소스 포함)
6. 웹브라우저는 받은 응답을 바탕으로 사용자에게 웹페이지 보여줌

### 16. HTTP/2를 설명하고 장점 두 가지를 설명하세요.

## 웹 주소 입력
17. www.naver.com을 주소창에 입력하면 어떻게 될까요?
1. host파일에서 도메인 네임(naver.com)에 대응하는 IP 주소가 있는지 확인, 없다면 네임서버에 요청
2. 받은 ip주소를 사용해 TCP 통신을 통해 해당 ip서버에 요청 보냄
3. 요청을 받은 서버는 요청 내용에 대한 일련에 처리 정을 거쳐 응답 메세지를 만듦
4. 응답메세지를 TCP 통신을 통해 다시 클라이언트에게 전송
5. 브라우저는 받은 응답메세지를 HTTP 프로토콜을 사용하여 웹페이지를 구성하여 사용자에게 naver 화면을 보여줌
